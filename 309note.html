WEEK5: jQuery, Node.js, AJAX, XML+JSON

jQuery

  • jQuery is based on JS and is still JS, it optimize to perform many common scripting functions using fewer lines of code=> interact with DOM => access and modify HTML using CSS-style selectors. Normally it would start with $(document).ready(function(){}); (load the function when DOM is ready), another choice is $(window).load(function(){}); (load event when DOM and all contents on the page(img, css, etc.) is fully loaded), $(document).ready might be master
  • Minified, un-minified; Minification => process that scripts out non-essential char from scripts, CSS, etc. make size smaller. Functionality not affected. Always keep a non-minified version
  • jQuery function
    • getters: read the section, operate on first element in a section
    • setters: change the section, operate on all elements in a section
  • $ is a shortcut for jQuery
  • There are a lot of events in jQuery, each corresponds to JavaScript
  • Use event handler .on() to interact with items that didn't exist when the DOM was loaded.
  • Event delegation => bind fewer event handlers than we'd have to if we were listening to clicks on individual elements
  • Recommanded: online tutorial about jQuery. This tutorial is very slow-paced but really go into details

NODE.JS

  • an event-driven, I/O model-based runtime enviroment and library for developing server-side web apps using JS. NODE.JS => real-time and fast
  • Modules => building block of Node.js and can be included by requiring in the main file. No need to include the .js file extension.
  • If you want another part of the web app to access variables and functions in a module, you have to expose these with exports.
    hello = function() {return “Hello world!”;}
    exports.hello= hell;
  • Recommanded tutorial: http://nodeschool.io
  • Recommanded: online tutorial about NODE.JS

AJAX

  • AJAX: Asynchronous JavaScript and XML, not a programming language, rather a way of using JS to download data from a server in the background. AJAX is a way of sending information between browser and server without refreshing page. It can be done with or without library like jQuery.

XML

  • data used on web in many ways: web servers/ databases/ web services

JSON(JavaScript Object Notation)

  • language-independent convention for formatting data as a set of JS obejcts
  • eg: 
    {"employees":[
    {"firstName":"John", "lastName":"Doe"},
    {"firstName":"Anna", "lastName":"Smith"},
    {"firstName":"Peter", "lastName":"Jones"}
]}

WEEK6: REST API; Database; ER-db-design

Rest APIs

  • URI(uniform resource identifier) => URL(universal resource locator), identifier for the location of document served by a web server
  • REST(Representational State Transfer): architectural style for using HTTP to provide resources over the web. REST api works pretty much the same way website does. Call from client to server => get data back over the HTTP protocol.
    • uniquely access resources through URLs
    • use GET, POST, PUT and DELETE
      • GET: reads specific resource or collection of resource(by identifier)
      • PUT: updates/ replaces a specific resource. Create if not exists
      • DELETE: remove a specific resource
      • POST: Creates a new resource. Use Post, not GET to make a change
    • use standard data format, HTML, XML, JSON
    • stateless protocol
  • 个人理解，REST API 是与网站做交互的另一种形式，使用REST API的methods比如GET/ PUT/ DELETE/ POST 可以直接访问网站数据，并达到发推，得知谷歌地图中特定位置的经纬的效果。REST 只是一种设计风格，而不是标准。 这种风格的特点为：
    • 直观简短的资源地址：URI，比如：http://example.com/resources/。
    • 传输的资源：Web服务接受与返回的互联网媒体类型，比如：JSON，XML，YAML等。
    • 对资源的操作：Web服务在该资源上所支持的一系列请求方法（比如：POST，GET，PUT或DELETE）。

Database

  • RDBMS: Relational Database Management System => persist data storage of large dataset s using a relational model.
    Relational => collection of table consists rows(for record) and columns(for attribute).
  • Data can be in relation of many-to-many and many-to-one
  • Pros of using RDBMS:
    • Data safety: immune to program crashes
    • Concurrent access: atomic access and updates
    • Data integrity
    • Fault tolerance
    • Scalability: handle large/ small quantities of data n a uniform manner
    • Reporting: write SQL to generate reports
  • More about SQL: clause/ expressions/ predicates/ queries/ statements
  • Data type:
    • document type => MongoDB
    • graph (in algorithm meaning) type => store info about networks, such as social connection and roadmaps
    • key-value type => every single item in DB stored as an attribute name( or "key") together with its value
    • column type => optimized for queries over large datasets and store columns of data together(?)
  • Relational databases require that schemas be defined before you can add data (?)
  • Relational databases usually scale vertically: a single server has to host the entire database to ensure reliability and continuous availability of data.

    • Entity Relationship Model

    • Modeling: about mapping entities and relationships from the world into a database
      Some concepts:
      • Entity: principal data object in DB. Entity set: class of objects that have properties in common.
      • Relationship: association between one or more entities. Cardinality => degree of a relationship
      • Attributes/ Composite attribute: desribe elementary properties of entities or relationships. Composite attribute: grouped attributes of the same entity or relationship have closely connected meaning or uses.
      • Cardinalities: Strength of connectivity, each entity set articipates in a relationship with a min and maxcardinality.
        Min cardinality = 0 means the entity participation is optional/ = 1 means participation is mandatory.
        Max cardinality = 1 means a 1-to-1 relationship/ = N means 1 instance of entity is associated with many instances of relationship. Attribute's cardinality:
      • Keys: minimal sets of attributes which uniquely identify instances of an entity set.
      • In constrast, a weak entity: cannot identified by attribute alone, rely on others.
    • Types of models
      • relational model: Use relations to represent the entities and relationships
      • visual data model: illustrates entities and relationships, as well as the attributes that describe them.

    Database schema design

    • Real World => The conceptual schema(E/R) => generate logical(relational) schema
      The translation: conceptual cschema -> logical model is not simple because not all of the constructs of ER model can be translated into relational model. Also the schema must be restructured.
      The translation can be done in two steps:
      1. Restruct of entity-relationship schema based on criteria for optimizing(?)
        *consider redundancy, generalization, entities and relations, limit weak entity sets, primary keys.
        • redundancy: repeated info or design => waste space, inconsistency, indicates design flaw. Decide to keep or eliminate based on the cost
        • Every relation must have a unique primary key and we tend to use weak entity sets (those without an internal primary key) when there’s no global authority that can create ID’s.
